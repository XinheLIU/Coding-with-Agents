# 如何与编程智能体协作

*本文提炼自 OpenAI 和 Anthropic 团队内部工作流——以及那些常让新手踩坑的失败模式。*

---

将 AI 视为简单代码补全工具的时代已经过去。当今的编程智能体——无论是 Anthropic 的 Claude Code 还是 OpenAI 的 Codex——早已不只是提供逐行建议的引擎。它们能读取你的文件、执行命令、修改代码、运行测试、提交 PR。它们可以运行在云端沙箱里，也可以直接驻留在你的终端中。它们能自主处理一项任务长达数小时。

这从根本上改变了协作的本质。问题不再是"AI 能写代码吗？"，而是：**如何与一个能够自主写代码的智能体协作？**

为了找到答案，我们直接追溯源头。Anthropic 和 OpenAI 都发布了详细报告，介绍各自内部工程团队如何使用这些工具。在 Anthropic，团队使用 Claude Code 处理从生产环境 Kubernetes 问题调试到让非技术财务人员通过自然语言生成 Excel 报表的各种场景。OpenAI 的 Codex 团队则报告，该工具编写了其内部应用代码的 90% 以上，新工程师入职第一天就与 Codex 结对，直接向生产环境交付代码。

> - [OpenAI 如何使用 Codex](https://cdn.openai.com/pdf/6a2631dc-783e-479b-b1a4-af0cfbd38630/how-openai-uses-codex.pdf)
> - [Anthropic 团队如何使用 Claude Code](https://www-cdn.anthropic.com/58284b19e702b49db9302d5b6f135ad8871e7658.pdf)

这两家公司呈现出高度一致的协作模式，并颠覆了许多关于 AI 辅助开发的传统假设。

## 核心洞察：渐进式自主权

Anthropic 和 OpenAI 最重要的核心原则，看似简单，却常被忽视：

> **对低风险、重复性、边缘性的工作授予高度自主权；对核心业务关键逻辑保持严格的人工监督。**

这听起来理所当然。但在实践中，大多数人恰恰做反了——他们对简单的事过度干预（盯着智能体写样板代码），却对难的事疏于监督（因为 AI"看起来很自信"就草草批准了复杂的架构变更）。

正确的心智模型是一条渐变的光谱，而不是一个非此即彼的开关。

### 高度自主的适用场景

两家公司一致将以下类别的工作委托给在后台异步运行、仅需松散监督的智能体：

**机械性变换。** 重构、重命名、跨文件迁移、更新废弃的 API 调用。OpenAI 工程师将大型迁移任务分配给多个并行运行的 Codex 实例——列出所有需要修改的文件，让智能体批量处理。Anthropic 团队在常规数据管道搭建中也有类似的模式。

**测试生成。** 编写单元测试、扩展边界情况覆盖、根据函数签名生成集成测试。Anthropic 研究团队报告，通过让 Claude Code 生成人类通常会跳过的边界情况测试，研究时间缩短了 80%。

**样板代码与脚手架。** 目录结构、模块桩代码、API 路由骨架、配置文件、遥测钩子。OpenAI 工程师使用 Codex 在开发早期搭建新特性的脚手架，并处理编写发布脚本等"最后一公里"任务。

**代码库探索。** 追踪数据流、梳理服务依赖、在陌生代码库中定位核心逻辑。Anthropic 数据团队用 Claude Code 取代了传统的数据目录工具——它能读取 `Claude.md` 文档、识别相关文件，并向新成员解释数据管道的依赖关系。

**内部工具与仪表盘。** Anthropic 的数据科学家在不懂 JavaScript 的情况下构建了超过 5000 行的 TypeScript 仪表盘。非技术背景的市场人员生成了数百个广告变体。律师搭建了电话树系统。模式非常清晰：智能体大幅拓展了"谁能构建有用软件"的边界。

**重复性运维工作。** Terraform 计划、K8s 日志分析、CI/CD 调试、基础设施即代码的安全审查。这些都是极佳的委托候选，因为其范围明确、模式固定，且在安全沙箱环境下风险较低。

### 限制自主权的适用场景

对于以下类别，智能体只应充当草稿提案者、结对编程伙伴或设计探索助手——而绝不能成为最终决策者：

- **核心业务逻辑** — 定义产品*做什么*的算法和规则。
- **安全敏感路径** — 身份验证、授权、数据处理、加密。
- **架构决策** — 系统边界、数据模型设计、API 契约。
- **高爆炸半径的变更** — 任何代价高昂、难以察觉或难以回滚的改动。
- **关键用户流程** — 影响信任与营收的核心流程。

Anthropic 明确指出：**同步结对**是处理核心业务逻辑的预期模式。智能体负责探索和提案，人类负责决策和验证。

### 决策框架

| 任务特征 | 自主权级别 | 工作模式 |
|---|---|---|
| 机械性、重复性、易于回滚 | 高 | 异步循环、批量处理、最少监督 |
| 探索性、面向设计、存在多条可行路径 | 中 | 生成多个方案供人工筛选（Best-of-N） |
| 核心逻辑、生产关键、安全敏感 | 低 | 同步结对、逐行 diff 审查 |

## 人工介入的时机

如果智能体负责处理"怎么做"——起草、搜索、重构——那么人类就必须把握"为什么"和"什么重要"。介入并非源于不信任，而是在必要之处运用人类判断力。

OpenAI 和 Anthropic 在四个关键介入点上高度一致：

### 1. 规划与范围界定

每一个高效的智能体工作流，都始于人类清晰地定义意图。两家公司都强调，在切换到代码生成模式之前，应先以"对话"或"询问"模式开始。你的提示词应像一份写得好的 GitHub Issue：目标明确、范围清晰、约束条件具体、相关参考完整、风格期望明确。

OpenAI 的最佳实践建议采用两步工作流：先请求一份**实施计划**，再将该计划作为代码生成的提示词。这能让智能体保持专注，避免"幻觉式范围蔓延"。

你在持久化记忆文件（Codex 使用 `AGENTS.md`，Claude Code 使用 `Claude.md`）中编码的上下文越丰富，智能体的表现就越好。Anthropic 数据团队发现了一个直接的正相关：随着 `Claude.md` 文件内容日益丰富，Claude Code 在日常任务中的可靠性也随之提升。这些文件将一个无状态工具转变为一个长期了解"游戏规则"的队友。

### 2. 核心逻辑与高风险变更

当工作进入不允许有误的领域时，协作模式必须从"委托"切换回"结对"。人类应当：

- 仔细审查每一个代码 diff（不要只看摘要）。
- 质疑假设——智能体可能以十足的自信犯错。
- 简化过度设计的方案（智能体倾向于复杂化）。
- 做出需要业务背景才能判断的权衡决策。

OpenAI 工程师描述，Codex 在处理人类大约需要一小时或几百行代码的任务时最为高效。对于更大规模的变更，"先规划再生成"的模式对于防止偏离至关重要。

### 3. 审查、合并与筛选

这一阶段由两种模式主导：

**Best-of-N 综合。** 让智能体为同一问题生成多个解决方案，比较各自的权衡，并综合最优的部分。输出的多样性能大幅提升复杂问题的质量——这是当前可用的最高杠杆协作模式之一。

**高频审查的 Git Flow。** 频繁提交，轻松回滚。两家公司都强调小步、频繁提交，以确保每一步都可见、可逆。OpenAI 内部团队通常同时运行 3-4 个完全独立的 Codex 任务，每个任务各自产出可检查的输出。

Anthropic 的 Boris Cherny 更进一步，引入了**审查子智能体**的概念：一个智能体检查代码风格，另一个审查项目历史，第三个追踪缺陷——同时还有更多智能体被分配去*反驳*这些发现，以过滤掉误报。

### 4. 不确定性与设计权衡

智能体非常擅长提供选项并压力测试你的假设。但决定接受哪种权衡——哪条架构路径契合你的产品路线图、团队能力和组织的风险承受能力——本质上是人类的工作。

将智能体视为**思考伙伴**。让它列出三种方案的优缺点。让它在你的代码库中寻找先例。让它找出某个选项中不可逆的部分。然后，由你来拍板。

### 责任归属模型

| 责任方 | 职责范围 |
|---|---|
| **人类** | 意图与目标 · 约束条件与风险评估 · 产品与业务判断 · 最终合并/部署/回滚决策 |
| **智能体** | 起草、搜索、重构 · 枚举选项与权衡 · 大规模批量重复工作 · 拓展范围与探索 |

## 7 个经过实战验证的协作模式

这些模式在 OpenAI 和 Anthropic 内部使用中反复出现，凝聚了数百小时工程实践经验，代表着真正有效的方法。

### 模式一：先规划，再生成

从对话开始。界定范围。就方案达成共识。只有这之后，才允许切换到批量代码生成模式。这能预防最常见的失败：一个令人印象深刻却解决了错误问题的方案。

### 模式二：小任务，频繁检查点

这与不信任无关——而是关于保持迭代速度。小步、已提交的进展意味着你能快速纠偏。两家公司都报告，这种模式显著降低了智能体犯错的代价，因为你距离一个已知良好的状态永远不超过一次提交。

### 模式三：持久化记忆文件

维护 `AGENTS.md`（Codex）或 `Claude.md`（Claude Code）文件，在其中编码项目的约定、规则、约束和架构决策。

这些文件将智能体从无状态工具转变为具备上下文意识的协作者。OpenAI 的 Codex 在执行任何工作前都会读取这些文件，并支持层级化系统——全局默认值、仓库级约定和目录级覆盖。

Anthropic 团队发现这种方式效果极佳，以至于它取代了其数据基础设施中的传统数据目录和可发现性工具。投入时间维护这些文件，回报远超预期。

### 模式四：边缘工作异步，核心工作同步

这也许是最重要的工作流洞察：**将你的人工介入程度与任务的重要程度相匹配。**

将边缘性工作（重构、测试、杂务、内部工具）委托给在后台运行的异步智能体循环。将你的同步注意力保留给核心逻辑、架构和关键路径。

OpenAI 工程师报告，他们在一天内合并了 4 个 PR，同时把大部分时间花在会议上——因为 Codex 一直在处理范围明确的后台任务。关键在于，这些都是经过适当范围界定的任务，而非核心逻辑。

### 模式五：Best-of-N 综合

对于存在多种有效方案的难题，生成多个解决方案并加以比较。这充分利用了智能体的速度，同时在最关键的地方应用了人类判断力。

请求同一功能的三种不同实现。分别请求性能最优版、可读性最佳版和可维护性最强版。然后从每个版本中汲取最优的想法加以融合。

### 模式六：智能体是力量倍增器，不是神谕

打断智能体。要求更简洁的版本。动态调整范围。在方案过度设计时提出异议。

正确的心智模型是一个迭代协作者，而非一次性的答案机器。最好的结果来自于将智能体输出视为通过对话不断打磨的初稿，而不是全盘接受或全盘否定的成品。

### 模式七：横向能力层

跨职能部署智能体，而不仅限于工程。来自两家公司最令人惊讶的发现，是这些工具被非工程师广泛使用的程度。

在 Anthropic，设计师进行大规模状态管理变更，市场人员生成广告变体，数据科学家用不熟悉的语言构建复杂仪表盘。在 OpenAI，Codex 团队构建了数百个横跨工程、产品、设计、数据和运营的可复用技能。

这意味着：编程智能体不只是开发者的生产力工具。它们是组织能力的倍增器，让领域专家无需排队等待开发，就能构建自己需要的东西。

## 常见失败模式（及规避方法）

了解有效模式还不够。你还需要识别出事情何时开始偏轨——最好在损失扩大之前。

### 需求层面的失败

**需求不清晰。** 让智能体构建"好用且高效"的东西，却不提供具体场景或验收标准，是白白浪费资源的配方。使用 Jobs-to-Be-Done 框架：用户是谁、他们想完成什么、你如何判断是否成功？

**功能复杂度爆炸。** 一开始就要求十几个功能，而不是专注的 MVP。智能体非常乐意构建你要求的一切——这让人很容易要求太多。将核心功能控制在 3-5 个，验证之后再扩展。

**功能脱离真实需求。** 从"全面的功能清单"出发，而不是分析真实用户工作流和现有替代方案。智能体可以构建任何东西；问题在于它是否应该这样做。

### 技术层面的失败

**技术选型不当。** 追逐新奇或"酷炫"的技术，而成熟、文档完善、对智能体友好的技术栈本可以更好地满足需求。智能体在拥有大量训练数据的主流框架上表现最佳。小众工具往往导致 API 幻觉和隐蔽的缺陷。

**过度依赖而不加验证。** "AI 说没问题就没问题"的失败模式。OpenAI 和 Anthropic 都将"信任但核实"作为基本操作原则。建立自己的判断框架。运行测试。阅读 diff。

**忽视基本功。** 只检查代码是否能运行，而不考虑安全性、性能或可维护性。智能体产出的代码可能通过所有测试，却存在 SQL 注入漏洞、本可以是 O(n) 的却写成了 O(n²)，或者采用了六个月后会带来痛苦的架构模式。

### 推广层面的失败

**目标用户过于宽泛。** "所有人都会用这个"不是一个用户细分。先服务好一个小群体，再考虑扩展。

**低估迁移成本。** 以为用户会因为你的方案技术上更优就主动切换。现有工具具有黏性——工作流、集成、肌肉记忆、团队培训。将每一个迁移步骤明确地梳理出来。

**高估持续参与度。** 你在蜜月期觉得一个工具好用，并不意味着用户会坚持使用它。在真实工作流中与真实用户进行足够长时间的测试。

## 实践入门指南

如果你是第一次采用编程智能体，以下是基于这些工具开发公司内部实践的分阶段方法：

**第一周：探索与学习。** 在对话模式下使用智能体来理解你的代码库。提出你在入职时会向资深工程师请教的问题。梳理依赖关系、追踪数据流、理解架构决策。

**第二周：委托边缘性工作。** 从测试生成、重构和文档编写开始。这些是高价值、低风险的任务，能帮助你建立对智能体能力的信心，以及审查其输出的能力。

**第三周：建立持久化上下文。** 编写你的 `AGENTS.md` 或 `Claude.md` 文件。在其中编码你的编码规范、测试要求、架构边界和领域专属术语。这是一次性的投入，对每次后续交互都会带来回报。

**第四周起：扩展至并行工作流。** 开始同时运行多个智能体任务。建立高效审查智能体输出的个人工作流。找出哪些任务通过异步委托能为你节省最多时间，哪些任务通过同步结对能产生最佳结果。

贯穿整个过程：频繁提交、仔细审查，并始终保持对代码做了什么、为什么这样做的独立理解。智能体是你判断力的倍增器——但前提是你持续发挥这种判断力。

## 展望未来

趋势已经明朗。OpenAI 设想实时结对与异步委托将融合为统一的工作流，让开发者能够大规模地指导、监督和协作智能体。Anthropic 的团队已经看到智能化编程超越传统开发范畴，演变为一种组织级的能力层。

在这种环境中脱颖而出的工程师，不会是那些最会写提示词的人。而是那些培养出强烈品味的人——能够快速评估、筛选和打磨智能体输出的能力。是那些在需求、约束和权衡上进行清晰思考的人。是那些根据任务重要程度调整介入深度的人，而不是事无巨细地干预一切，也不是盲目信任一切。

智能体负责苦活累活，你负责判断。这就是协作的本质。
